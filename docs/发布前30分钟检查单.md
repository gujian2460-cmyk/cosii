# 发布前 30 分钟检查单（Cosii API）

面向 **单次上线窗口** 的可执行清单；与 `docs/go-live-checklist.md` 长期项互补。  
证据建议：每条保留命令输出片段或截图，附日期与环境名。

## T-30：门禁复跑（本地或 CI）

```bash
npm run verify:gates
```

等价于顺序执行：`npm test` → `npm run test:critical` → `npm run test:e2e`。

## T-25：目标环境探活（需服务已监听）

默认本机 `http://127.0.0.1:3000`：

```bash
npm run dev
```

另开终端：

```bash
npm run verify:ready
```

预发/生产替换基址：

```bash
set API_BASE=https://你的预发域名
npm run verify:ready
```

确认：`/health` 与 `/v1/ready` 均为 **200**，JSON 含 `code: "OK"` 与 `trace_id`。

## T-20：结构化访问日志（代码已支持，发布环境留样）

1. 进程环境设置：`STRUCTURED_ACCESS_LOG=1`
2. 对任意路径发 1～2 次请求（如 `GET /health`）
3. 在标准输出中确认 **单行 JSON**，字段含：`trace_id`、`path`（或等价 URL）、`status_code`，且 **无** 密钥/Authorization 原文

若使用容器/平台日志，确认日志聚合里能按 `trace_id` 检索。

## T-15：配置与密钥（生产项）

对照 `docs/go-live-checklist.md` **配置与密钥** 全节；未就绪则 **禁止切流量**。

重点：`DB_PATH` / `DATA_DIR`、`LOCAL_CODE_PEPPER`、微信支付回调 URL 与证书、生产禁用纯 `X-User-Id` 信任边界；`WECHAT_APPID` / `WECHAT_SECRET` / `COSII_SESSION_SECRET`；**真机**至少走通一次 `wx.login` → `/v1/auth/wechat-login` → Bearer 业务请求（证据见 `go-live-checklist` 对应勾选项）。

## T-10：回滚

阅读并确认一人可执行：`docs/runbooks/发布回滚步骤.md`。

- 探活 200（LB → `/health`）

- 记录首笔请求的 `trace_id` 备查
