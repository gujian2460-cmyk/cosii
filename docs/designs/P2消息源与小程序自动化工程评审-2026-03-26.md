## 0. Step 0 — Scope Challenge

## 0.1 What already exists

1. **消息页可运行**：`miniprogram/pages/inbox/inbox.js` 已有 loading/error/列表与深链跳转。  
2. **当前消息源是“订单列表映射”**：消息数据来自 `GET /v1/me/unified-orders?limit=5`，非独立通知模型。  
3. **小程序自动化尚未接入**：仓库内没有 `miniprogram-automator` 或真机回归脚本文件。  
4. **后端门禁已齐**：`verify:gates` 对 API 覆盖完整，但不覆盖小程序运行时交互。

## 0.2 最小可交付（MVP）

- **P2-1 独立通知源（后端 + 前端）**
  - 新增后端通知表与 API：`GET /v1/me/notifications?cursor=&limit=`
  - `inbox.js` 从 unified-orders 切换到 notifications API
  - 保留 `unified_order_id/order_type/domain_order_id` 作为深链上下文
- **P2-2 小程序自动化**
  - 先落“可在 CI 跑通”的 Node 脚本级冒烟（真机脚本可后补）
  - 至少覆盖：主页加载、订单 tab 切换、消息页列表渲染、通知点击跳转参数

## 0.3 Complexity Check

- 预计涉及文件 > 8（后端 schema/routes/service + 小程序页面 + 测试脚本 + CI）。
- 结论：采用**两阶段分批**最稳妥：
  1) 通知源改造；2) 自动化脚本接入。

## 0.4 Completeness Check

- 只改前端消息来源而不建独立通知模型，会把“消息语义”和“订单摘要语义”继续耦合。  
- 只写自动化脚本不稳定注入数据，会得到易波动的假阳性结果。  
- 建议做完整闭环：**通知数据建模 + 稳定种子 + 自动化脚本 + 门禁接入**。

## 1. Architecture Review

## 1.1 目标架构（建议）

```text
业务事件（支付/发货/争议/结算）
          |
          v
   notification_events (DB)
          |
          v
 GET /v1/me/notifications
          |
          v
 inbox page list -> navigateTo(order-detail)

自动化层：
seed fixture -> mini-program smoke script -> CI gate
```

## 1.2 架构问题（按严重度）

1. **高**：消息源复用订单列表会丢失“事件类型/已读状态/去重键”。  
2. **中**：无自动化导致 UI 改动依赖手测，回归成本高且易漏。  
3. **中**：若通知 API 不携带稳定深链字段，消息点击会出现跳转失败或参数缺失。

## 1.3 Opinionated Recommendation

- 建议采用“**独立通知表 + 读接口 + 轻量写入点**”模式，不直接从 `unified_orders` 反推消息。  
- 自动化先做“稳定冒烟”，再逐步增加真机回归；避免一开始重投入但不可维护。

## 2. Code Quality Review

## 2.1 当前问题

1. `inbox.js` 与订单列表耦合，字段语义不是“消息”。  
2. 消息标题目前统一硬编码“订单状态更新”，缺失事件类型驱动渲染。  
3. 自动化体系缺位，后续 UI 重构难保回归质量。

## 2.2 建议落地方式（最小 diff）

1. 后端新增 `src/modules/notification/{routes,service}.ts`，只暴露读接口。  
2. 在支付/争议/核销等关键路径中补“消息落库”函数（可先 3 个事件类型）。  
3. `inbox.js` 保持现有 UI，替换 data source 为 notifications API（降低前端风险）。  
4. 新增 `scripts/miniprogram-smoke.mjs`（含固定数据种子与基础断言）。

## 4. Performance Review

1. 通知列表应按 `created_at DESC` + 索引（`user_id, created_at`）分页。  
2. 每页默认 `limit=20`，并限制最大值（如 50）。  
3. 消息内容体避免大字段（正文、富文本）直接放列表接口，详情按需取。

## 5. Failure Modes

| 失败模式 | 测试覆盖 | 错误处理 | 用户可见 |
|---|---|---|---|
| 通知 API 回落到订单映射导致重复/漏事件 | GAP | 无统一保护 | 易静默错误（高） |
| 通知缺少深链字段，点击失败 | GAP | 仅 fallback 到订单 tab | 可见（中） |
| 自动化脚本依赖不稳定环境，CI 偶发失败 | GAP | 可通过固定种子缓解 | 可见（中） |

**Critical gap**：通知模型未独立，无法保证“消息=事件”的真实性。
