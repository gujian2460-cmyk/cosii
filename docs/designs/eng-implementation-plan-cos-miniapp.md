## State Machines

### Trade Order

```text
DRAFT
  -> PENDING_PAYMENT
  -> PAID_ESCROW
  -> SHIPPED
  -> DELIVERED
  -> CONFIRMED
  -> SETTLED

Side paths:
PENDING_PAYMENT -> CANCELLED
PAID_ESCROW -> DISPUTED -> (REFUNDED | RESUMED -> SHIPPED)
```

### Service Order

```text
DRAFT
  -> SLOT_HELD
  -> DEPOSIT_PAID
  -> CONFIRMED_BOOKING
  -> SERVICE_IN_PROGRESS
  -> SERVICE_COMPLETED
  -> FINAL_PAID
  -> SETTLED

Side paths:
SLOT_HELD -> EXPIRED
CONFIRMED_BOOKING -> RESCHEDULE_REQUESTED -> (APPROVED | REJECTED)
ANY_ACTIVE -> DISPUTED -> (REFUND_PARTIAL | REFUND_FULL | RESUME)
```

### Dispute

```text
OPEN -> EVIDENCE_COLLECTION -> UNDER_REVIEW -> DECIDED -> CLOSED
                \-> TIMEOUT_ESCALATED
```

## State Dictionary (single source of truth)

| State ID | Domain | Meaning | Allowed Next States | User Label |
|---|---|---|---|---|
| `PENDING_PAYMENT` | trade/service | 等待支付 | `PAID_ESCROW`, `CANCELLED` | 待支付 |
| `PAID_ESCROW` | trade | 已完成托管支付 | `SHIPPED`, `DISPUTED` | 已托管 |
| `DEPOSIT_PAID` | service | 已支付定金 | `CONFIRMED_BOOKING`, `DISPUTED` | 定金已支付 |
| `SERVICE_COMPLETED` | service | 服务已完成待尾款 | `FINAL_PAID`, `DISPUTED` | 待支付尾款 |
| `DISPUTED` | trade/service | 争议处理中 | `REFUNDED`, `RESUMED`, `DECIDED` | 仲裁处理中 |
| `SETTLED` | trade/service | 已完成结算 | terminal | 已结算 |

Rule: UI labels and state transitions must reference this table only.

### State authority rule

- `unified_orders.status` is the authoritative cross-domain state for listing/filtering/settlement eligibility.
- Domain tables (`trade_orders`, `service_orders`) keep domain-specific substates only.
- Any state transition must update `unified_orders` + domain row in one transaction.
- Forbidden: async dual-write without transaction boundary.

## Error UX Mapping

| Error Code | User Title | Primary Action | Retry Policy |
|---|---|---|---|
| `PAYMENT_TIMEOUT` | 支付结果确认超时 | 查看订单状态 / 联系客服 | auto retry status sync x3 |
| `PAYMENT_WEBHOOK_DUPLICATE` | 支付结果已同步 | 返回订单详情 | no user retry needed |
| `BOOKING_SLOT_CONFLICT` | 该时段已被占用 | 重新选择时段 | immediate re-pick |
| `DISPUTE_INVALID_STATE` | 当前状态无法发起该操作 | 查看争议流程说明 | no retry, guided path |
| `SETTLEMENT_RETRY_EXHAUSTED` | 结算处理中，请稍后查看 | 打开订单结算明细 | background retry + support fallback |
| `CONTENT_CARD_TARGET_INVALID` | 关联内容已失效 | 浏览其他可购内容 | no retry needed |

## Processing Pipelines (ASCII)

### Payment to Settlement

```text

  -> Ordering Guard (provider event time/sequence monotonic check)
  -> Mark PAID_ESCROW/DEPOSIT_PAID
  -> Emit SettlementPending -> Settlement Job
  -> Ledger Events (freeze -> commission -> release)
```

### Booking Concurrency

```text
Select Slot -> Create Hold(2m) -> Tx Begin -> Check Unique Constraint
  -> Lock Slot Row -> Create Service Order -> Commit
  -> Hold Released/Consumed
```

## Performance Plan

- Composite indexes delivered in initial migration.
- Queue partitioning:
  - `payments-webhook`
  - `settlement-jobs`
  - `notifications`
- Retry policy:
  - exponential backoff (max 5)
  - dead-letter on exhaust
- Read cache (phase 1):
  - artist slots (short TTL)
  - item detail summary
- No cache for payment/dispute state (must be source-of-truth reads).

### Performance safeguards added in this rerun

- Slot cache invalidation: write-through invalidate on slot hold/create/expire transitions.
- Query discipline:
  - order list endpoints must preload payer/dispute summary to avoid N+1.
  - post detail must batch-load conversion card targets by type.
- Queue protection:
  - settlement worker concurrency cap with per-order key serialization.
  - dead-letter replay tool must enforce idempotency-key reuse.

## Financial Reconciliation Acceptance Criteria

- Daily reconciliation must satisfy:
  - `sum(settlement_ledger.net_settled)` == `sum(provider_settled_amount)` within tolerance `<= 0.1%`.
  - mismatch above tolerance triggers P1 alert and blocks next settlement batch.
- Reconciliation report fields (required):

- Out-of-order callbacks must not regress final state once terminal transition committed.
