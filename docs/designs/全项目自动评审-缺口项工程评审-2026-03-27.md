## 1. Architecture Review

### 1.1 数据流（目标态）

```text
微信登录 → session → 小程序 storage / header
              ↓
发布闲置 → POST /v1/trade/items（或既有创建接口）→ 列表 / 详情
发布作品 → POST /v1/posts (+ cards) → 内容详情
约妆入口 → POST /v1/booking/orders 等 → 订单 / 支付
订单列表 → GET /v1/me/unified-orders?order_type=&status=&cursor=
```

### 1.2 主要问题（按严重度）

1. **高 — 身份边界：** 生产仍依赖 `X-User-Id` 时，「我的」与所有写接口在真机无安全边界；架构上必须先定义 **session → user_id** 的单一注入点（middleware / `api.js`），再改页面。
2. **中 — 分包入口空壳：** booking/content 入口无路由到具体表单页，用户从 Tab 或「我的」点入后**认知断裂**（无下一步）。
3. **中 — 发布无写路径：** `publish` 只有导航，没有与后端写模型对齐的 **单一入口服务层**（易在多个页面重复 `request` 拼装）。
4. **低 — 筛选维度不全：** `status` 已在后端存在；前端不传则**无法用同一列表页做「待付款/进行中」运营视图**。

### 1.3 建议结论

- **身份：** 先定「小程序侧 token / code2session 结果存哪、谁负责刷新」再动 UI。
- **发布：** 优先 **trade listing 一条链**（字段少、接口现成度高），content/booking 跟其后。
- **分包：** entry 页至少 **重定向或内嵌第一步表单**，避免空白 `onLoad`。

## 2. Code Quality Review

### 2.1 观察

1. **DRY：** `orders.js` 的 `buildOrdersPath` 与 `list.js` 的 `buildTradeListPath` 模式一致，后续若加更多筛选，可抽 **小工具函数**（仅限 query 拼装），避免过早抽象「通用列表框架」。
2. **显式优于聪明：** `me.js` 中 `isLoggedIn` 与 `devUserId` 关系应在注释或常量中写清「生产必须移除 dev header」。
3. **占位页技术债：** `packageBooking` / `packageContent` 的 `entry` 无文案与返回路径时，**调试体验差**；至少保留与设计稿一致的说明 + `navigateBack`。

### 2.2 建议

- 发布表单落地时：**校验与 API 错误码**复用 `utils/errors.js`，不新开一套 toast 文案。
- 登录流程：**单文件封装** `login()` / `ensureSession()`，禁止每个页面各自 `wx.login` 碎片逻辑。

## 4. Performance Review

1. **列表分页：** `unified-orders` / `trade/items` 已 limit+cursor；发布成功后列表 **应 refresh 第一页**，避免全量重拉无分页的旧数据。
2. **发布图片：** 若后续接本地选图上传，需在架构层预留 **压缩与直传**（否则性能与成本问题在小程序端 first paint）。
3. **分包懒加载：** entry 空壳页无重逻辑，性能无风险；真正风险在 **首屏请求数**（onShow 多次 `refresh` 已与订单页类似，注意去抖或缓存策略）。

## 5. Failure Modes

| 场景 | 测试 | 错误处理 | 用户可见 |
|------|------|----------|----------|
| 未登录调写接口 | 部分 API 测 | 401 + envelope | 应引导登录；现为 toast/占位 |
| 发布字段校验失败 | GAP | 依赖后端码 | 需表单 inline 错误 |
| booking 空入口 | 无 | 无 | **空白页级体验（中高）** |
| 仅完成 order_type 未做 status | 无 | N/A | 运营视图缺失（中） |

**Critical gap（延续原报告精神）：** 生产仍信任 `X-User-Id` 时，任何「我的页闭环」都是**假闭环**。
