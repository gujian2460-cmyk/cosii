## 评审范围

| 能力域 | 规划要点 |
|---|---|
| 内容发布 | 发布作品/图文内容；产出可被购买转化的内容入口 |
| 转化卡 | 在内容页挂接商品/服务卡片；目标对象失效时安全降级 |
| 同城核销 | 线下交付场景的核销码流程（创建、校验、完成、失败回退） |

**真源对齐：**
- 端点规划：`POST /v1/posts`、`POST /v1/posts/{id}/cards`
- 错误码：`CONTENT_CARD_TARGET_INVALID`
- 履约关联：`trade_fulfillments`（`ship_mode`, `proof_url` 等）
- 顺序节点：`Implementation Sequence` 中 Day7-8（content + city verification）

## Phase 1 - CEO 评审（范围与策略）

### 结论

- Day7-8 的目标不是“做一个社区”，而是“打通内容到成交与线下交付闭环”的最小可用链路。
- 应避免把推荐分发、复杂内容治理、直播化能力塞进本阶段（超出 Day7-8 范围）。

### 战略风险

1. **转化卡目标一致性风险**  
   内容关联对象（商品/服务）下架或失效后，若仍可点击下单，会造成错误订单路径。

2. **核销安全风险**  
   核销码若无过期时间、尝试次数、状态机约束，容易被重放或撞库。

3. **流程割裂风险**  
   内容卡跳转、下单、履约、核销若不串联同一订单上下文，运营无法追溯漏斗。

### CEO 通过条件

- Day7-8 仅交付：
  - 内容发布接口与最小内容实体能力
  - 转化卡挂接与失效降级
  - 同城核销码完整状态流转（含失败与重试边界）
- 其他能力（推荐引擎、直播电商、跨平台分发）保持 deferred。

## Phase 2 - Design 评审（UI/UX）

### 设计要求（与 `DESIGN.md` 一致）

1. **内容详情页首屏必须有清晰主行动**  
   转化卡可见且主次按钮明确，不允许“信息很多但无法行动”。

2. **卡片失效必须优雅降级**  
   目标失效时显示“内容已失效/已下架”并给替代路径，不展示无效 CTA。

3. **核销流程状态可读**  
   至少覆盖：待核销、核销成功、核销失败、已过期，且文案+颜色双通道表达。

4. **异常态不黑盒**  
   网络失败、核销码过期、重复核销必须给用户可恢复动作（重试/返回订单/联系客服）。

### Design 分数（0-10）

| 维度 | 分数 | 说明 |
|---|---:|---|
| 信息架构 | 8 | Day7-8 目标清晰，内容到转化链路明确 |
| 状态完整度 | 7 | 需补齐核销失败细分状态与文案 |
| 行动可达性 | 8 | 转化卡作为主转化入口合理 |
| 异常可恢复性 | 7 | 需明确失败后主按钮策略 |

## Phase 3 - Eng 评审（架构与测试）

### 建议实现边界

- **内容发布**
  - `POST /v1/posts` 支持基本发布能力与鉴权
  - 对内容字段做长度、类型、空值校验

- **转化卡**
  - `POST /v1/posts/{id}/cards` 建立内容到目标对象的映射
  - 目标对象校验失败返回 `CONTENT_CARD_TARGET_INVALID`
  - 读取端对失效对象自动降级隐藏或置灰

- **同城核销**
  - 建立核销码生命周期：`PENDING -> VERIFIED -> COMPLETED/EXPIRED/FAILED`
  - 核销动作要求幂等，重复提交不产生二次履约
  - 与订单履约表（含 `trade_fulfillments`）保持一致事务边界

### 测试覆盖图（P0）

| 路径 | 测试类型 | 预期 |
|---|---|---|
| 发布内容成功/参数非法 | 契约 + 集成 | 200 / 400；返回 envelope 合规 |
| 创建转化卡目标有效/无效 | 契约 + 集成 | 无效目标返回 `CONTENT_CARD_TARGET_INVALID` |
| 内容目标下架后访问 | 集成 | 卡片自动降级，不可下单 |
| 核销码首次/重复核销 | 集成 | 首次成功，重复请求幂等 |
| 核销码过期 | 集成 | 返回过期状态与可恢复动作提示 |
| 并发核销 | 集成 | 仅一次成功，其他请求安全失败 |

### 工程风险与约束

- 核销码需有 TTL 与尝试上限，防止暴力尝试。
- 关键操作写入可审计字段（trace_id、operator、timestamp）。
- 不引入新协议或新服务拆分，保持 Day7-8 可交付节奏。

## 决策审计摘要（/autoplan 原则）

| # | 原则 | 本轮决策 |
|---|---|---|
| 1 | 完整性优先 | 不只做 happy path；强制包含失效、过期、并发核销 |
| 2 | 煮沸湖面 | Day7-8 变更覆盖内容、转化卡、核销及对应测试 |
| 3 | 务实 | 不扩到推荐系统与直播模块 |
| 4 | DRY | 复用统一 envelope、错误码、状态字典 |
| 5 | 显式优于聪明 | 核销状态机与 TTL 采用显式字段表达 |
| 6 | 偏向行动 | 先交付最小闭环，再扩运营能力 |

## 通过标准（写入日终验收）

1. `POST /v1/posts`、`POST /v1/posts/{id}/cards` 可用并通过契约测试。  
2. 转化卡目标失效时返回/展示行为与 `CONTENT_CARD_TARGET_INVALID` 对齐。  
3. 同城核销流程具备幂等与过期控制，并覆盖并发场景。  
4. 关键异常态（失效、过期、重复核销）均有可恢复 UX。  
5. `npm run build` 与 `npm test` 通过。
