## 2. 前提与边界（CEO / 产品）

以下前提若变更，需回写本文与 `miniprogram-ui-shell-spec.md`。

1. **设计稿代表目标体验**：粉紫梦幻、大圆角、TDesign 风格组件；主色与工程当前 `--color-primary: #D878A0` 一致。  
2. **业务仍以 Cosii 后端契约为准**：交易 `GET/POST /v1/trade/*`、约妆 `packageBooking`、内容 `packageContent`；设计稿中的「租/借」「置顶」「价格区间」等需 **API 或配置层支持** 才能真数据，否则保留 Mock + 占位。  
3. **微信 Tab 限制**：原生 TabBar **中间大「+」** 需自定义 TabBar 或单页容器 + 假 Tab；有实现成本与审核/交互风险，需在 IA 决策中单独立项。  
4. **DESIGN.md 深色舞台规范**与小程序浅色稿并存：支付结果、争议、SLA 等 **状态色与文案** 仍以 `DESIGN.md` Day3–8 为准，避免仅用粉色替代语义色。

**未纳入本次对标范围（建议写入 TODOS）**

- 完整 TDesign npm 引入与按需构建链（可后续统一）。  
- 所有设计稿中的真实图片 CDN、域名白名单、内容审核策略。

## 3. 信息架构对照（IA）

| 设计稿 Tab | 当前 `app.json` | 差距 |
|------------|-----------------|------|
| 首页 | `pages/home/home` | 模块部分对齐，缺「最近漫展」按钮、5 金刚文案、热门角色、精选返图叠字样式等。 |
| **买&搜**（合并） | **购买** `buy` + **搜索** `search` **分两 Tab** | **不一致**：需合并为单 Tab 或保留双 Tab 但 UI 上模拟「买&搜」容器页。 |
| **中间 +** | **发布** `publish`（普通 Tab 项） | **不一致**：设计为凸起圆形主按钮；需自定义 TabBar 或首页 FAB。 |
| 消息 | **无独立 Tab**（在「我的」内 `navigateTo` inbox） | **不一致**：设计稿为独立 Tab + 角标能力。 |
| 我的 | `pages/me/me` | 布局与视觉与稿差异大（稿为渐变头 + 多组宫格）。 |

**决策建议（自动推荐：完整性优先）**

- **推荐 A**：实现 **自定义 TabBar**（或微信自定义 tabBar 组件），五段与设计稿一致；中间「+」打开发布分流或半屏菜单。  
- **备选 B**：维持 5 Tab 但文案改为「买&搜」+ 内嵌 `switch` 到搜索子视图，消息拉回 Tab（仍无中间凸起，接受视觉降级）。

## 6. 工程（Eng）

### 6.1 依赖与路由

- 新增页面建议放在 `miniprogram/pages/` 或分包（按体积）：`expo/`（漫展）、`recent-expo/`、`work-detail/`、`seller/`、`buy-search/`（合并容器）。  
- `app.json` 的 `tabBar.list` 与 `pages` 数组需与 IA 决策同步更新。  
- 自定义 TabBar 时：**中间「+」** 与 **消息角标** 需单独逻辑（`badge` 可用接口轮询或 WebSocket，后续定）。

### 6.2 与后端映射（CLAUDE.md）

| 设计功能 | 建议 API / 说明 |
|----------|-----------------|
| 商品列表筛选排序 | `GET /v1/trade/items` 需扩展 query（价格区间、排序、区域）或前端过滤（仅 Mock 阶段）。 |
| 作品流 / 返图 | `GET /v1/posts` 或专用 feed（若未就绪则 Mock）。 |
| 卖家主页统计 | 聚合 `trade` + `posts` + 评价（若未有则 Mock）。 |
| 漫展 | 可能为 **纯运营配置** 或新模块；需产品定数据模型后再接 API。 |

### 6.3 测试与门禁

- **Vitest**：不涉及小程序 UI。  
- **`npm run smoke:miniprogram`**：仍应通过；若改 `inbox`/`orders` 引用需更新 `scripts/miniprogram-smoke.ts`。  
- **建议新增**：对关键 `Page` 路径与 `request` path 的静态扫描或最小 E2E（可选）。

## 10. 工程维度速评

- **架构**：当前分包（trade/booking/content）与稿中「多频道电商+社区」可兼容；宜增加 **轻量「频道配置」** 驱动金刚区与分类，避免硬编码。  
- **重复**：`buy` 与 `packageTrade/list` 逻辑重复已有先例；合并「买&搜」时应 **抽公共请求层**，避免三份列表逻辑。  
- **安全**：卖家主页、作品详情上线前需确认 **用户隐私与鉴权**（与 `CLAUDE.md` Bearer / 生产头一致）。
