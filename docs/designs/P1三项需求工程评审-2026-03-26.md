## 0. Step 0 — 范围挑战（Scope Challenge）

## 0.1 What already exists（已有能力）

1. **订单列表基础链路已可用**
   - 前端 `pages/orders/orders.js` 已接 `/v1/me/unified-orders`，支持 `cursor` 分页。
   - 后端 `src/modules/me/routes.ts` + `src/modules/me/service.ts` 已提供统一订单列表与单条摘要。
2. **订单详情深链已可用**
   - 前端可 `navigateTo` 详情分包；后端已有 `GET /v1/trade/orders/:tradeOrderId` + booking 详情接口。
3. **请求与错误处理基线已具备**
   - `miniprogram/utils/api.js` + `miniprogram/utils/errors.js` 已建立统一入口。

## 0.2 三项 P1 的最小改动集（推荐）

1. **订单筛选联调（P1-1）**
   - 在 `GET /v1/me/unified-orders` 增加查询参数：`order_type`、`status`；前端 `onFilterTap` 真正带参请求。
2. **我的页登录与角色态（P1-2）**
   - 增加 `GET /v1/me/profile`（返回登录态、昵称、角色、头像）。前端 `pages/me/me.js` 根据返回切换 UI。
3. **在售列表闭环（P1-3）**
   - 增加 `GET /v1/trade/items`（分页 + 可选筛选），`packageTrade/pages/list/*` 改为真实列表与进入详情/下单入口。

## 0.3 Complexity check（复杂度）

- 预计会触及 **>8 文件** 且新增 **>2 服务函数**，属于中等复杂度。
- 结论：**建议分三波实现**（见 §6 实施顺序），避免一次性高风险改动。

## 0.4 Completeness check（完整性）

- 若只做前端筛选 UI 不做后端 query，会导致分页与筛选语义不一致（伪完成）。
- 若只做“我的页文案切换”不接真实登录态，会把身份问题延后到上线阶段（高风险）。
- 若在售列表只做 mock，会影响闭环验证（首页 → 列表 → 详情/下单）。
- **结论：应做完整版（端到端）而非半截版。**

## 1. Architecture Review（架构评审）

## 1.1 数据流与边界

```text
[orders page filter click]
        |
        v
GET /v1/me/unified-orders?order_type=&status=&cursor=&limit=
        |
        v
[unified_orders + joined trade/service aggregates]
        |
        v
orders list (filtered, cursor-stable)

[me page onShow]
        |
        v
GET /v1/me/profile
        |
        v
login state + role badge + menu visibility

[home/publish -> trade list package]
        |
        v
GET /v1/trade/items?cursor=&limit=&category=&price_min=&price_max=
        |
        v
trade list cards -> trade detail -> create order
```

## 1.2 架构问题（按严重度）

1. **高：订单筛选必须在后端实现，不能只前端过滤**
   - 原因：cursor 分页基于后端排序，前端过滤会破坏“下一页”一致性。
2. **高：我的页角色态依赖真实认证边界，不应继续使用纯开发态假身份**
   - 原因：上线前会出现“显示角色与后端权限不一致”风险。
3. **中：在售列表目前为占位页，首页主 CTA 的业务闭环不完整**
   - 原因：核心转化路径被阻断，影响真实验收与指标采集。

## 1.3 Recommendation（架构）

- 保持“薄客户端 + 后端聚合”的现有路线，不新增复杂中间层。
- API 统一继续走 envelope，不引入第二套错误协议。

2. `me.js`
   - 统一 `data: { loading, error, isLoggedIn, profile }`。
   - `onShow -> refreshProfile`，失败统一 `mapEnvelopeToError`。
3. `trade list.js`
   - 复用 `request + mapEnvelopeToError`，实现 `refresh + loadMore + onFilter` 三段式。

## 4. Performance Review（性能评审）

1. 订单筛选与分页必须走后端索引查询，避免前端整页拉取后过滤。
2. 在售列表需控制一次 `setData` 体积；分页默认 `limit` 建议 20。
3. 价格区间筛选建议后端进行整数比较（分/厘），前端只做显示格式化。

## 5. Failure Modes（失败模式）

| 失败模式 | 测试覆盖 | 错误处理 | 用户感知 |
|---|---|---|---|
| 前端筛选但后端不支持，分页结果错乱 | GAP | 部分 | 可能静默错误（高） |
| 我的页角色态与后端权限不一致 | GAP | 部分 | 页面显示与实际权限冲突（高） |
| 在售列表接口超时 | GAP | 可复用现有 error toast/页内状态 | 可见（中） |
| 在售列表空态文案缺失 | GAP | 无 | 可见但体验差（低） |

**Critical gap：** 第一项（筛选分页语义错乱）若不上测试，容易在生产出现“看见的不是实际数据窗口”。
