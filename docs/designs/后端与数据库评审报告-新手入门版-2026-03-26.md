## 4) 新手该怎么做（一步一步照做）

下面给你一个“从今天到能独立维护后端”的最短路径。

## 第 0 步：把项目跑起来（先赢一次）

在 PowerShell 执行：

```powershell
npm install
npm run dev
```

另开一个终端：

```powershell
npm run verify:ready
```

看到 `/health` 与 `/v1/ready` 正常即通过。

## 第 1 步：学会看一条请求完整链路

建议从这条链路看：

1. 路由入口：`src/modules/trade/routes.ts`
2. 业务逻辑：`src/modules/trade/service.ts`
3. 表结构：`db/schema.sql`（`trade_items`、`trade_orders`、`unified_orders`）
4. 测试：`tests/orders.contract.test.ts`

目标：你能回答“一个下单请求从哪进、查了哪些表、如何返回、失败怎么报错”。

## 第 2 步：先做一个小改动并带测试

新手第一改动建议：

- 给某个 GET 接口加一个安全的 query 参数（如 `limit` 上限保护）。
- 同时补一条 Vitest（成功 + 参数非法）。

做完后跑：

```powershell
npm test
```

## 第 3 步：掌握数据库安全三件套

每次改 SQL 都检查：

1. 是否参数化（避免拼字符串注入）。
2. 是否有索引支撑查询条件（尤其列表与分页）。
3. 是否保证幂等（支付/回调/重试路径）。

## 第 4 步：把“上线前必做项”变成固定流程

每次准备上线前固定执行：

```powershell
npm run verify:gates
```

并按文档逐项打勾：

- `docs/go-live-checklist.md`
- `docs/发布前30分钟检查单.md`

## 6) 给新手的防坑清单（最实用）

1. 不要直接改线上数据库文件；先本地验证 + 备份策略。
2. 不要跳过测试直接“感觉没问题”；至少跑 `npm test`。
3. 不要在路由里写复杂业务；把逻辑放到 service 层。
4. 不要返回随意 JSON；统一用 envelope（`code/message/data/trace_id`）。
5. 不要把“暂时方案”带到生产（尤其 `X-User-Id` 和 mock 开关）。
