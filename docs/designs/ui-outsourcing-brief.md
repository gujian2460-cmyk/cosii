# Cosii 小程序 UI 外包设计方案（委托版）

## 1. 目标与背景

- 产品：`Cosii`（微信小程序），核心场景为交易、约妆、内容转化。
- 本轮目标：由外部设计师产出一套可直接落地开发的 UI 设计稿与规范，不做业务重定义。
- 设计基线：必须遵守 `DESIGN.md` 与 `docs/designs/miniprogram-ui-shell-spec.md`。

## 2. 设计原则（必须遵守）

### 2.1 视觉方向

- 风格：Retro-Futuristic + Editorial（舞台感 + 杂志排版），不是通用 SaaS 模板。
- 默认深色主题（单主题冻结）：背景 `#0B0F14`，表面 `#111827`，主文 `#F9FAFB`。
- 主行动色：`#E11D48`；信息/链接色：`#0EA5E9`。

### 2.2 交互硬约束

- 每屏仅一个主 CTA，禁止并列双主按钮。
- 禁止首屏三列等权功能卡网格。
- 状态表达必须“文案 + 颜色”双编码，不能只靠颜色区分。
- 支付、档期、争议、核销等关键状态必须一眼可读，且有明确下一步动作。

### 2.3 信息架构

- 五 Tab：`首页` / `发布` / `订单` / `消息` / `我的`。
- `订单` 是状态真源；`消息` 只做提醒与深链，不复制完整时间线。

## 3. 外包范围（In Scope）

1. 小程序五个 Tab 页的高保真视觉稿（含深色主题）。
2. 核心组件规范：按钮、状态条、卡片、输入框、骨架屏、金额行。
3. 关键状态页：支付结果、档期倒计时、争议与 SLA、核销状态、内容转化卡可用/不可用态。
4. 页面级交互说明：点击、禁用、加载、空态、错误态、超时态。
5. 可交付开发的标注与切图规范。

## 4. 不在本轮范围（Out of Scope）

- 新增业务流程、字段定义、后端接口改造。
- 双主题（浅色模式）设计。
- 品牌重塑、Logo 重做、营销活动专题页。

## 5. 交付清单（必须）

1. 设计源文件（Figma，含可编辑组件库）。
2. 页面稿：
   - 五 Tab 主页面
   - 关键流程页面（支付/档期/争议/核销/内容转化）
   - 至少覆盖 `加载 / 空 / 错误 / 成功 / 超时` 五类状态
3. Design Tokens 文档（颜色、字体、间距、圆角、阴影、动效时长）。
4. 组件规范文档（状态、尺寸、可用性、禁用规则）。
5. 交互说明文档（页面跳转、按钮优先级、异常路径文案）。
6. 开发对接文档（标注、切图、命名、导出倍率、图标网格规则）。

## 6. 里程碑与节奏（建议）

- M1（第 2 天）：风格板 + 首页/订单两页初稿评审。
- M2（第 5 天）：五 Tab 主页面 + 核心组件库 v1。
- M3（第 8 天）：关键状态页补齐（支付/争议/核销/内容）。
- M4（第 10 天）：全量交付包 + 开发走查 + 修订定稿。

## 7. 验收标准（给甲方/产品/前端）

### 7.1 结构与转化

- 首页首屏存在且仅存在一个主 CTA。
- 信息层级清晰：先信任与状态，再行动入口，再次级内容。

### 7.2 状态完整性

- 每个关键流程都有 `加载/空/错误/成功`。
- 对“超时、可疑、过期、已同步、不可用”有专门状态表现，不混用。

### 7.3 一致性

- 颜色、间距、圆角、文案语气统一。
- 同类组件在不同页面状态一致（禁用态、错误态、反馈时长一致）。

### 7.4 可开发性

- 设计中每个关键元素均可映射到小程序组件实现，不依赖不可落地效果。
- 提供明确标注，不出现“看着做”或含糊字段。

## 8. 外包协作注意点（重点）

1. **先锁框架后抛光**：先确认 IA、页面关系、状态完整性，再优化视觉细节。
2. **先做状态页，不只做正常页**：支付超时、核销过期、争议逾期是高风险页面，必须前置设计。
3. **避免“好看但不可做”**：小程序端避免复杂混合模糊、超重动效、大面积视频背景。
4. **文案不是占位符**：关键按钮和状态文案要贴近真实业务语义，避免“确定/提交”泛文案。
5. **图标需统一系统**：线宽、拐角、填充规则统一，避免套用多个图标风格。
6. **把异常路径当主路径设计**：用户最关心失败后怎么办，必须有明确下一步。
7. **版本管理清晰**：每轮评审输出 changelog，避免“视觉回退”与口头变更丢失。

## 9. 常见风险与规避

- 风险：只交静态页面，无组件规范。  
  规避：验收必须包含组件库与状态表。

- 风险：把消息页做成第二套订单系统。  
  规避：明确“消息只提醒 + 深链，订单才是真源”。

- 风险：双主按钮导致转化分散。  
  规避：每屏单主行动，次操作降级为文字链。

- 风险：关键状态视觉权重混乱（处理中像失败）。  
  规避：按语义色与状态条规则统一，输出状态矩阵。

- 风险：外包稿脱离现有设计真源。  
  规避：评审 checklist 强制逐条对照 `DESIGN.md` 与 UI shell spec。

## 10. 给外部设计师的执行说明（可直接复制）

> 你将负责 Cosii 微信小程序的 UI 完整设计交付。  
> 请严格基于现有设计真源，不重定义业务流程。  
> 目标是“可直接开发落地”，不是纯视觉展示。  
> 设计需覆盖五个 Tab、核心组件、关键状态页和异常路径，输出可标注可导出的完整交付包。

## 11. 评审 Checklist（开会直接用）

- [ ] 是否严格遵守 `DESIGN.md`（颜色、层级、单主 CTA、反 AI slop）  
- [ ] 是否覆盖五 Tab 主页面  
- [ ] 是否覆盖关键状态（支付/档期/争议/核销/内容可用性）  
- [ ] 是否包含组件规范与状态矩阵  
- [ ] 是否提供开发标注与资源导出规范  
- [ ] 是否定义异常路径的用户可见下一步  
- [ ] 是否能在小程序技术约束内实现
