## 1. Architecture Review

### 1.1 条 1 — 契约测试层

```text
wx.response → parseHttpResult(status, body) → { ok, data?, envelope? }
                     ↑
              Vitest（Node，无微信运行时）
```

- **建议：** 将 `api.js` 中 `success` 回调内「从 status + body 到 `{ ok, … }`」抽为 **`parseMiniprogramResponse(status, body)`**（纯函数，无 `wx`）。`request()` 仅胶水。
- **契约快照：** 对典型 OK / fail 信封 JSON 做 `toMatchSnapshot()` 或与 `src/shared/api-envelope` 类型断言（字段子集）。
- **风险：** 双真源 — 若后端改信封字段，需同时更测试；可接受。

### 1.2 条 2 — 错误体验

- **现状：** 各页复制 `loading / error / else` 与 `cos-page-state__retry`。
- **建议：** 微信小程序可用 **template + slot** 或单一 **behavior**（`behaviors/page-state.js`）减少 WXML 重复；**或**保留现状仅统一 `app.wxss` 中 `.cos-page-state` 变体。
- **不推荐：** 过早上自定义组件库（创新 token 超支）。

### 1.3 条 3 — 可观测性

- **架构：** 日志链 `trace_id`（HTTP）与业务表 `trace_id`（如 settlement）已部分存在；缺的是 **人读文档**。
- **产出物：** `docs/runbooks/trace-id-playbook.md` 一节即可：示例 grep/查询语句、与 `unified_order_id` 对照。

### 1.4 条 4 — 发布护栏

- **架构：** CI 或 `npm run build:miniprogram` 前执行 `node scripts/check-miniprogram-prod.mjs`：`readFile app.js` → 若匹配 `useMockOrders:\s*true` 且 `process.env.COSII_MINIPROGRAM_ENV=production` 则 exit 1。
- **备选：** 仅检查上传目录不含 `useMockOrders: true` 字符串（简单 grep）。

### 1.5 条 5 — 文档与 routes 一致

- **架构：** 脚本扫描 `app.get/post/...` 注册路径 vs `CLAUDE.md` 中 `` `GET /v1/...` `` 列表；**允许**「文档多写说明性路径、脚本只警告不失败」的第一版。

## 3. Test Review

### 3.1 覆盖图（针对五条）

```
[1] parseMiniprogramResponse / parseEnvelope
    ├── 2xx + code OK        → [GAP → Vitest]
    ├── 2xx + code !OK       → [GAP]
    ├── 2xx + 非对象 body    → [GAP]
    ├── 4xx + JSON envelope  → [GAP]
    └── fail 合成 NETWORK    → [GAP]（可模拟不调 wx）

[2] 页面错误条               → [GAP] 仍以手测为主；抽模板后可 snapshot WXML（可选）

[3] runbook                    → [N/A] 人工验收

[4] check script               → [GAP] Vitest exec 或自脚本 --dry-run 自测

[5] doc vs routes              → [GAP] 脚本 + fixtures
```

### 3.2 建议测试与落地方式

- **条 1（推荐路径）：** 新增 `miniprogram/utils/parse-response.js`，导出纯函数 `parseWxEnvelopeResult(statusCode, body)`（无 `wx` 依赖）；`api.js` 的 `success` 回调仅调用该函数再 `resolve`。Vitest 用 `import ... from "../miniprogram/utils/parse-response.js"`（或 `createRequire` 加载 CJS）覆盖：2xx+OK、2xx+业务错、2xx+非信封、4xx+JSON 信封、非 2xx 无 JSON。**禁止**在测试中复制粘贴与 `api.js` 两套逻辑（易漂移）。
- **契约快照：** 对 2～3 个典型 `body` 使用 `toMatchInlineSnapshot` 或固定 `expect(result).toEqual({...})`。
- **条 4：** 对 `scripts/check-miniprogram-prod.mjs` 编写小型「夹具 `app.js`」临时文件 + `execSync`，断言 exit code。
- **条 5：** 脚本自身可用 Vitest 测「提取路径列表」正则；或先人工维护 `expectedPaths[]` 对齐 golden list。

## 4. Performance Review

- **条 1：** 纯函数抽离对运行时无负面影响；`request` 多一次函数调用可忽略。
- **条 2：** Behavior/template 若引入额外 `setData` 需避免双层刷新；默认无问题。
- **条 5：** 全仓库扫描 `routes.ts` 在 CI 中应控制在约 1 秒内；若变慢再缓存或只扫 `src/modules`。

## 5. Failure Modes

| 条 | 失败模式 | 测试 | 用户可见 |
|----|----------|------|----------|
| 1 | 抽函数后与 `api.js` 行为不一致 | Vitest 必须与改前行为对齐 | 错误 toast/页态错误（**高**） |
| 2 | 统一模板后某页漏绑 `refresh` | 手测清单 | 重试无效（中） |
| 3 | runbook 过时 | 无自动 | 排障慢（中） |
| 4 | 脚本漏检（字符串变形绕过） | 加反面用例 `useMockOrders : true` | mock 上生产（**高**） |
| 5 | 正则漏路由或误报 | allowlist | CI 噪音或假安全（中） |

**Critical gap：** 条 4 未做时，与「生产构建」流程无硬闸，仅靠人工 — 与自动评审原文一致。
