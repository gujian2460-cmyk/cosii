# 发布回滚步骤（Cosii MVP / SQLite）

适用于 **API 单体现状**；数据库为 SQLite 时默认 **前进 only**，回滚版本不等于回滚数据，需单独备份/恢复策略。

## 1. 停流量

- 负载均衡 / 网关：将后端实例从池摘除，或切换维护页。
- 确认无新用户请求进入旧版本实例（连接Drain）。

## 2. 回退运行版本

- 容器/平台：回滚到上一稳定镜像或上一部署版本号。
- 裸机：`git checkout <tag>` 或解压上一发布包 → `npm run build && npm start`（按你方部署脚本为准）。

## 3. 数据库决策（SQLite）

- **仅代码回滚、无迁移变更**：通常 **无需** 动库文件；确认 `DATA_DIR`/`DB_PATH` 指向与回滚前一致。
- **本次发布含 schema 变更且已写脏数据**：
  - 若有 **发布前备份**：按运维手册恢复 `app.db`（及 `-wal`/`-shm` 若使用 WAL）。
  - 若无备份：评估是否接受手工修复或保持新版本库 + 仅回滚代码（可能不兼容）— **高风险**，发布前应禁止此状态。

## 4. 结算 / 异步任务

- 若存在自动结算、队列消费者：回滚前记录开关状态；回滚后确认 **不会双扣/双结**（幂等键与 job 状态以代码与库为准）。
- 具体开关名以部署配置为准；无全局开关时，以「停流量 → 排空队列 → 再回滚」为保守顺序。

## 5. 回滚后验证

```bash
# 指向回滚后的基址
set API_BASE=https://你的环境
npm run verify:ready
```

再抽样一条主业务链（与 `docs/发布前30分钟检查单.md` 一致）。

## 6. 记录

- 回滚原因、操作人、时间点、涉及版本号、是否动库，写入发布说明或值班工单。
